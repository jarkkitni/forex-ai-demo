# เทคนิค "นกน้อยพิมพ์รัง" — เปิดร้านใหม่จาก Lullabell Template

> ตั้งชื่อ 21 ก.ค. 2026 — แนวคิด: เราสร้างบอทให้ Lullabell (ร้านทำผม) จนพิสูจน์แล้วว่าใช้งานได้จริง
> แทนที่จะเริ่มจากศูนย์ทุกครั้งที่ได้ลูกค้าใหม่ ให้ "พิมพ์" (stamp) โครงเดิมออกมาซ้ำๆ
> เปลี่ยนแค่ข้อมูลร้าน (config) — โค้ดหลัก (meta_bot.py, ai_guard.py) ใช้ตัวเดียวกันทั้งหมด ไม่ต้องเขียนใหม่

## สรุปสั้นๆ ก่อน — ตอบคำถามที่ถามมา

**"ทำมันขึ้นมาอีกได้เรื่อยๆ แค่ขอข้อมูลจากลูกค้าไหม?"**
ได้ — แต่ **ต้องเลือกวิธีก่อน** เพราะโค้ดตอนนี้ (21 ก.ค. 2026) ยังเป็นแบบ "ร้านเดียวต่อ 1 ระบบ" (single-tenant)
ไม่ใช่ "หลายร้านในระบบเดียวกัน" — รายละเอียดอยู่หัวข้อ "ทำไมโคลนไม่ได้ตรงๆ" ด้านล่าง

**"ต้องไปสมัคร Facebook App ใหม่ทุกร้านไหม?"**
**ไม่ต้อง** — App "NexUs-SiriAriyaMate Chat" ที่มีอยู่แล้วใช้ซ้ำได้กับทุกร้าน (ดูหัวข้อ "ฝั่ง Facebook/Meta")
สิ่งที่ต้องทำซ้ำต่อร้านคือ **เพิ่ม Page ใหม่เข้า App เดิม** ไม่ใช่สร้าง App ใหม่

**"ลูกค้าแอดเราเป็นแอดมิน แล้วเราจัดการที่เหลือได้เลยไหม?"**
ได้ — นี่คือขั้นตอนจริงเป๊ะที่ทำกับ Lullabell (ดูขั้นตอน 4-6 ด้านล่าง) ไม่มีขั้นตอนพิเศษอื่นแอบซ่อนอยู่

## ทำไมโคลนโฟลเดอร์ตรงๆ ไม่พอ (สิ่งที่ต้องรู้ก่อนรับร้านที่ 2)

เช็คโค้ดจริงแล้ว (21 ก.ค. 2026): `api_server.py` route `/webhook/meta` ตอนนี้ยิง event ทุกอันไปที่
**ค่าเดียวตายตัว** — `META_PAGE_TOKEN`, `META_SLUG` (default "lullabell"), `META_PAGE_ID` — เป็น env var เดี่ยวๆ
ไม่มีการเช็คว่า event นี้มาจาก Page ไหนแล้วเลือก config/token ให้ตรงร้าน

**แปลว่า:** ถ้าเอาร้านที่ 2 มาเสียบใน deployment เดียวกับ Lullabell ตอนนี้เลย — ข้อความจากทั้ง 2 ร้าน
จะไปปนกัน (ร้าน B อาจได้คำตอบราคาของร้าน A) เพราะระบบไม่รู้จักแยก

### ทางเลือก 2 แบบ

**แบบ A — Deploy แยกต่อร้าน (ใช้ได้เลยวันนี้ ไม่ต้องแก้โค้ด)**
ก๊อป repo นี้ทั้งก้อนไปตั้งเป็น Render service ใหม่ ใส่ env var ชุดใหม่ (token/slug/page id ของร้านใหม่)
ข้อดี: ไม่มีความเสี่ยงกระทบร้านเดิม, ทำได้ทันที
ข้อเสีย: ทุกร้านมีค่าใช้จ่าย/ดูแลแยกกัน (แต่ Render free tier ยังไหวถ้าไม่เยอะมาก), โค้ดซ้ำกันหลายก้อน แก้บั๊กต้องไล่แก้ทีละที่

**แบบ B — Multi-tenant ในระบบเดียว (ต้องเขียนโค้ดเพิ่ม ยังไม่ได้ทำ)**
เพิ่มตาราง page_id → slug/token ให้ `/webhook/meta` เช็คก่อนว่า event มาจาก Page ไหนแล้วเลือก config/token ให้ถูก
ข้อดี: รับร้านใหม่ = แค่เพิ่มข้อมูล ไม่ต้อง deploy ใหม่ทุกครั้ง, แก้บั๊กที่เดียวกระทบทุกร้าน (เหมือนที่เพิ่งแก้ bug 1,000 ตัวอักษรวันนี้)
ข้อเสีย: ต้องเขียนโค้ดก่อนใช้ได้ (ใช้เวลา 1 รอบงาน) — token ของแต่ละร้านต้องเก็บใน DB/config แทน env var เดี่ยวๆ

**คำแนะนำ:** ร้านที่ 2 (ยังไม่มีตัวจริง) ใช้แบบ A ไปก่อนได้เลย ไม่ต้องรอ
ถ้าเริ่มมี 3-4 ร้านขึ้นไปแล้วดูแลลำบาก ค่อยขอให้สร้างแบบ B (บอกได้เลย จะเข้า Plan Mode ตามขั้นตอนปกติ)

## ฝั่ง Facebook/Meta — อะไรทำครั้งเดียว vs อะไรต้องทำซ้ำทุกร้าน

**ทำครั้งเดียว (เสร็จแล้ว ใช้ซ้ำได้ทุกร้าน):**
- App "NexUs-SiriAriyaMate Chat" (App ID 1558681259271673) — ไม่ต้องสร้างใหม่
- Toggle field "messages" ในหน้า Messenger API Settings → Webhooks — เป็นการตั้งค่าระดับ App ไม่ใช่ระดับ Page
- หน้า Privacy Policy / Data Deletion (`/privacy`, `/data-deletion`) — ใช้ร่วมกันได้ทุกร้าน
- App ผ่านสถานะ "เผยแพร่แล้ว" (published) — ไม่ต้องขอใหม่

**ต้องทำซ้ำทุกร้านใหม่:**
1. ลูกค้าเพิ่มเรา (sIRImeta) เป็น **แอดมิน/Editor** บน Facebook Page ของร้าน (เหมือนที่ Lullabell ทำ)
2. ถ้ามี Instagram ด้วย ต้องเป็น IG Professional Account ที่ผูกกับ Page นั้นแล้ว
3. ไปที่ Meta for Developers → App เดิม → Graph API Explorer → ขอ token จาก `/me/accounts`
   จะเห็น Page ใหม่โผล่มาในลิสต์ (เพราะเราเป็นแอดมินแล้ว) → แลกเป็น **long-lived Page Token** (ไม่มีวันหมดอายุ)
4. เรียก API subscribe webhook ให้ Page ใหม่: `POST /{page-id}/subscribed_apps?subscribed_fields=messages,messaging_postbacks,message_reactions`
5. ถ้าจะส่งรูปแผนที่ร้าน (attachment) — ยิง `/api/admin/warm-attachment` **2 รอบ**: รอบแรกเปล่าๆ (ใช้กับ FB)
   รอบสองใส่ `?platform=instagram` (ใช้กับ IG — ไม่ทำรอบนี้ รูปจะขึ้น FB แต่ไม่ขึ้น IG เหมือนที่เจอกับ Lullabell)
6. (ถ้าทำ LINE OA ด้วย) สร้าง LINE Official Account ของร้านนั้นแยก ได้ channel token/secret มาใส่ env

**เรื่อง Business Verification:** Business "Jarkkit" ที่ผูกกับ App ยังไม่ผ่าน Meta Business Verification
(เช็คแล้ว 21 ก.ค. 2026) — ตอนนี้ยังใช้งานได้ปกติเพราะ permission ที่ใช้อยู่ (`pages_messaging` ฯลฯ) ไม่ได้บังคับ
verify ในทุกกรณี แต่ถ้าจำนวนร้าน/ข้อความเยอะขึ้นมาก Meta อาจขอให้ verify — เผื่อใจไว้เป็นงานในอนาคต

## ข้อมูลที่ต้องขอจากลูกค้าใหม่ (เอาไปกรอก `configs/_TEMPLATE.json`)

ส่งลิสต์นี้ให้ลูกค้ากรอก หรือถามเป็นข้อความก็ได้:

1. **ข้อมูลร้าน:** ชื่อร้าน (ไทย+อังกฤษ), ประเภทธุรกิจ, สโลแกน (ถ้ามี), อีโมจิประจำร้าน
2. **ช่องทางติดต่อ:** เบอร์โทร, LINE OA ID, ชื่อเพจ FB, username IG, ที่อยู่เต็ม, ลิงก์ Google Maps, จุดสังเกตใกล้ร้าน, ที่จอดรถ
3. **เมนู/ราคาเต็ม:** แยกเป็นหมวดหมู่ พร้อมราคาทุกรายการ (รูปแบบตารางเมนูจริงที่ลูกค้าใช้อยู่ ก็อปมาแปลงได้เลย)
4. **โปรโมชั่นปัจจุบัน:** อันไหนคือโปร (ต้องรู้ชัดว่าอันไหน "hot" เพราะบอทจะดึงเฉพาะกลุ่มนี้ไปตอบตอนลูกค้าถามหาโปรโดยตรง)
5. **ของแถม/สิทธิพิเศษ:** ลูกค้าใหม่ได้อะไร, มาใช้บริการทุกครั้งได้อะไรแถม, โปรชวนเพื่อน
6. **โทนการพูด:** อยากให้ AI แทนตัวเองว่าอะไร (เช่น "น้องเบลล์") พูดสุภาพ/เป็นกันเอง/ทางการแค่ไหน
7. **เคสที่ต้องส่งต่อแอดมินจริง:** มีเงื่อนไขไหนที่ห้าม AI ตอบเอง (เช่น แพ้ง่าย/ตั้งครรภ์/เคสร้องเรียน) ต้องส่งต่อคนจริงเสมอ
8. **รูปผลงาน:** ถ้ามีรูปผลงานอยากโชว์ (ไม่บังคับ)
9. **Facebook/Instagram access:** ขอให้ลูกค้าเพิ่ม sIRImeta เป็นแอดมิน/Editor บน Page ของร้าน

## ขั้นตอนทำจริง (แบบ A — deploy แยก, ใช้ได้วันนี้)

1. ก๊อป repo `forex-ai-demo` ทั้งก้อนไปตั้ง Render service ใหม่ (หรือถามก่อนว่าจะแยก repo จริงๆ ดีกว่าไหม
   ถ้าจะดูแลหลายร้านระยะยาว — โค้ดส่วนที่ไม่เกี่ยวกับบอทแชท เช่น Forex/Hunter/Dialysis เอาออกได้ ลดความซับซ้อน)
2. ก๊อป `client-template/configs/_TEMPLATE.json` → `configs/{slug}.json` แล้วกรอกข้อมูลจากลิสต์ด้านบน
3. อัปโหลดรูปแผนที่/ผลงานไป `/static/{slug}/`
4. ทำขั้นตอนฝั่ง Facebook ด้านบน (ข้อ 1-6) ให้ครบ
5. ตั้ง env vars ชุดใหม่ให้ service ใหม่: `META_PAGE_TOKEN`, `META_SLUG={slug}`, `META_PAGE_ID`, `META_APP_SECRET`,
   `META_VERIFY_TOKEN`, และถ้าทำ LINE ด้วย: channel token/secret ของร้านนั้น
6. ทดสอบทักจริงทั้ง 3 ช่องทาง (LINE/FB/IG) ก่อนส่งลูกค้า — เช็ค Render logs ว่า deploy ขึ้นจริงก่อนเทส (บทเรียนจาก Lullabell)
7. ทำ closing message ปิดงานส่งลูกค้าตามแบบที่ทำให้คุณ Eye (ดู order #49XOM4U4 เป็นตัวอย่าง)
